1. Executive Summary

The three-layer pipeline proposed in the Week 5 deliverable is now fully implemented and validated on real dog videos. The system correctly detects dogs, computes motion, applies adaptive sampling, and produces structured behavioral assessments , end-to-end, from raw video to JSON + HTML report , without manual intervention.

Result: All three test videos produced correct breed identifications, valid 7-category behavioral assessments, and functional HTML timeline reports. Adaptive sampling reduced VLM calls by 25–75% depending on video activity level, directly controlling the largest cost driver in the pipeline. The codebase is modular, tested (66/66 unit tests pass), and architecturally ready for cloud deployment.

This document covers: (1) what was built and how it maps to the Week 5 design, (2) the real-video validation results, (3) the evaluation strategy needed before model quality can be claimed, (4) every tunable parameter and what it controls, and (5) the complete cloud deployment architecture the client should begin preparing.

2.2 The Three Layers

Layer 1
Low-Level Vision
YOLOv8-nano dog detection → padded bounding-box crop  |  Farneback dense optical flow → normalized motion score [0–1]  |  Activity classification: static / transition / active / no_detection
↓
Layer 2
Adaptive Sampler
Stateful per-video decision engine , sliding motion window → decision: analyze / keyframe / skip  |  Priority rules: no-dog → skip; motion spike → keyframe; active → always analyze; static → 1-in-N periodic sample
↓
Layer 3
VLM Analysis
VLMAnalyzer sends dog crop (not raw frame) to Ollama qwen2.5vl:7b  |  Returns 7-category behavioral JSON (tag + intensity 0–10)  |  Breed identification on first detected frame  |  vLLM swap point: _call_backend()
↓
Output
PipelineOutput dict , fully JSON-serializable, returned by run_pipeline() with no file I/O  |  Callers (CLI / FastAPI / SQS worker) own persistence and reporting

2.3 Key Design Decisions

DecisionRationale
run_pipeline() is a pure function Returns a dict, writes nothing. CLI, FastAPI, and SQS workers all call it identically. Enables direct await loop.run_in_executor(None, run_pipeline, path) in async contexts.
Dog crop sent to VLM (not raw frame) Tighter subject view; removes background distractors. Layer 1 encodes the crop to JPEG only after YOLO runs, so no wasted encoding on frames that get skipped.
Optical flow between sampled frames One flow computation per sample interval , same cost as Track D's frame extraction, no per-raw-frame iteration needed. Sufficient for activity-state classification.
Dependency injection on VLM + sampler Both objects are injectable into run_pipeline(). Tests pass mock objects; production passes real ones. No monkey-patching required.
TypedDicts, not Pydantic Zero additional runtime dependencies. Dicts are natively JSON-serializable. Pydantic adds value once the schema stabilizes post-evaluation (Week 9–10 upgrade candidate).
All config is env-var overridable Every threshold, URL, model name, and path in shared/config.py has an os.environ.get() fallback. Docker deployments set values via environment , no code changes needed.

3.1 Adaptive Sampling in Action

The sampling decisions for test1 illustrate the system working as intended. The mostly-still dog triggered periodic static sampling and one keyframe when motion spiked:

t=0 s  ANALYZE forced (first frame , breed ID)
t=2 s  skip   motion=0.051, static, interval 1/3
t=4 s  ANALYZE static periodic sample (3/3)
t=6 s  skip   motion=0.080, static, interval 1/3
t=8 s  skip   motion=0.086, static, interval 2/3
t=10 s  KEYFRAME motion spike 0.176 (> 2× window avg)
t=12 s  skip   motion=0.114, static, interval 1/3
t=14 s  skip   motion=0.133, static, interval 2/3

For test2 and test3, the consistently high motion scores (≥0.45) caused the sampler to classify nearly every frame as active, analyzing them all , which is correct behavior for an energetic dog.

3.3 What We Cannot Claim Yet

These results demonstrate functional correctness, not model accuracy. We can confirm that the pipeline runs, detects dogs, classifies motion, and produces plausible VLM outputs. We cannot yet claim that the behavioral tags are quantitatively accurate because we have no labeled ground truth to compare against. That is the primary goal of Weeks 8–10 (see Section 4).

4. Next: Evaluation Strategy & Ground Truth

The single most important thing to do next is collect a small labeled dataset and measure how well the pipeline performs. Without this, we cannot make meaningful claims about model quality, cannot justify parameter choices, and cannot set a baseline for future fine-tuning work.

4.1 What Videos to Collect

Aim for 20–40 short clips (10–30 s each) representing diverse behavioral scenarios. Priority coverage:

Behavioral scenarioWhy it mattersTarget count
Calm / resting dog (low arousal)Establishes baseline "relaxed" ground truth; tests static sampling6–8 clips
Playful / active dogTests active-state sampling; should trigger near-100% analysis rate6–8 clips
Anxious / fearful cues (ears back, tail tucked, yawning)Critical clinical signal; most likely to expose VLM errors4–6 clips
Pain indicator cues (guarding, orbital tightening, reluctance)Highest-stakes category; must not be over- or under-reported4–6 clips
Camera motion / dog partially out of frameTests YOLO robustness and flow score inflation (known limitation)3–4 clips
Multiple dogs in sceneConfirms the "highest-confidence detection only" policy is safe2–3 clips

Clips should include at least 3 different breeds and ideally both indoor and outdoor settings. Existing client footage is ideal if available , even 10 labeled clips are enough to get a preliminary accuracy signal.

4.2 Ground Truth Annotation Protocol

Each clip needs frame-level labels for the 7 behavioral categories. Recommended approach:

  1. Annotator: One veterinary behaviorist (or experienced vet tech) is sufficient for v1 evaluation. For training data at scale, a 3-annotator majority vote with inter-rater agreement (Cohen's κ) will be needed.
  2. Tool: Export the pipeline's sampled frames to JPEG (using --save-frames flag), then annotate in any spreadsheet or lightweight labeling tool (CVAT, Label Studio, or even Google Sheets). One row per sampled frame, one column per behavioral category.
  3. Label format: For each category, record (a) the correct tag from the enum and (b) an intensity score 0–10. This directly matches the pipeline's output schema , comparison is a simple string match + MAE.
  4. Minimum viable set: 200–400 labeled frames across the 20–40 clips. At 2-second intervals with an average clip length of 15 s, this means roughly 150 frames per 10 clips.

4.3 Metrics to Compute

5. Tunable Parameters & What They Control

All parameters live in shared/config.py and are overridable by environment variable. No code changes are needed to experiment , values can be changed at runtime or in a .env file. The table below covers every parameter that meaningfully affects output quality or cost.

5.1 Layer 1: Detection & Motion

ParameterDefaultEffect of increasingEffect of decreasing
YOLO_CONFIDENCE0.40 Fewer detections; reduces false positives in crowded scenes More detections; may include partial or low-quality crops
YOLO_MODEL_SIZEyolov8n.pt (nano) Larger model (s/m/l) → better detection accuracy, slower per-frame Nano is already fastest; no smaller option
CROP_PADDING_PX20 px More context around dog for VLM; may include more background Tighter crop; VLM sees less context (tail/ear edges may be clipped)
FLOW_MAGNITUDE_SCALE15.0 px/frame Motion scores decrease , more frames classified as static (less VLM cost, may miss transitions) Motion scores increase , more frames classified as active (more VLM cost, better coverage)
MAX_IMAGE_DIMENSION720 px Higher resolution input to VLM; more VRAM required and slower Faster, less VRAM; fine detail (eye whites, ear position) may be lost

5.2 Layer 2: Adaptive Sampling

ParameterDefaultEffect of increasingEffect of decreasing
MOTION_THRESHOLD_STATIC0.05 More frames classified as static → more skipping Fewer frames classified as static → more VLM calls
MOTION_THRESHOLD_ACTIVE0.15 Harder to reach "active" state → more frames stay in transition Easier to reach "active" → more VLM calls during moderate motion
STATIC_SAMPLE_EVERY_N3 Lower sample rate during static periods → fewer VLM calls, sparser timeline Higher rate → denser coverage of static periods (diminishing returns)
MOTION_WINDOW_SIZE5 frames Slower transition detection (spike must exceed longer average) More reactive to short bursts; may produce false keyframes on noise
DEFAULT_INTERVAL_SECONDS2.0 s Sparser sampling; faster processing; may miss short behavioral events Denser sampling; better temporal resolution; more VLM cost

5.3 Layer 3: VLM

ParameterDefaultNotes
VISION_MODELqwen2.5vl:7b Swap to qwen2.5vl:72b or a fine-tuned specialist for higher tag accuracy. The biggest single lever for quality improvement , evaluation results in Weeks 8–10 will determine if this is necessary before deployment.
TEMPERATURE0.1 Low temperature keeps outputs deterministic and consistent. Increase only if you want to sample multiple responses and aggregate (ensemble strategy).
MAX_RETRIES2 Total retry attempts on VLM failure before returning placeholder. Raise on unreliable network/hardware; lowering speeds up failure recovery.

5.4 Improvement Levers (Ranked by Expected Impact)

  1. VLM model upgrade: 7B → 72B parameter model is likely the largest accuracy gain available without any training data. Requires GPU with ~48 GB VRAM (A6000 or 2×A100) or a hosted API.
  2. Prompt engineering: The behavioral assessment prompt in shared/schema.py can be refined based on evaluation error patterns. If the VLM consistently confuses ears erect and ears neutral, adding explicit disambiguation language to the prompt is faster than fine-tuning.
  3. Sampling interval tuning: Run the same video at 1 s, 2 s, and 3 s intervals and compare behavioral timeline completeness against ground truth. Set per-video-type defaults (e.g. shorter interval for clinical assessment videos).
  4. Fine-tuning on domain data: Once 500+ labeled frames are available, fine-tune the VLM on dog behavioral assessment. Expected gain: +10–20 percentage points on tag accuracy for the hardest categories. Planned for Weeks 9–11.
  5. Dog-crop-only optical flow (v2): Currently flow is computed over the full frame, inflating motion scores when the camera moves but the dog is still. Restricting flow computation to the YOLO bounding box eliminates this artifact.
  6. Temporal smoothing across frames (v2): Tag outputs currently treated independently per frame. Exponential moving average (EMA, α=0.3) across consecutive analyzed frames would reduce flickering between adjacent tags and make mental state timelines smoother.

6. Cloud Deployment Architecture

The pipeline was designed from the start with production deployment in mind. run_pipeline() is a pure function , it accepts a file path and returns a dict. Every integration point (FastAPI, SQS, vLLM, Supabase) is a thin wrapper around that function, not a rewrite. Below is the target architecture for Weeks 11–14.

Client App (web/mobile) │ POST /analyze (multipart video upload, or presigned S3 URL) ▼ ┌─────────────┐ │ FastAPI │ , validates request, writes video to Supabase Storage │ (AWS EC2 │ , enqueues job message to AWS SQS │ or ECS) │ , returns job_id immediately (async response) └──────┬──────┘ │ SQS message: { job_id, video_s3_uri, options } ▼ ┌─────────────────────┐ │ Pipeline Worker │ , downloads video from S3 to /tmp │ (EC2 G-instance │ , calls run_pipeline(local_path) │ or ECS + GPU) │ , saves PipelineOutput to Supabase Postgres └──────┬──────────────┘ (frames table + jobs table) │ ▼ ┌────────────┐ ┌──────────────────┐ │ vLLM │ │ Supabase │ │ (GPU node)│ │ · Storage: raw │ │ qwen2.5vl │ │ videos + HTML │ │ 72B │ │ · Postgres: │ └────────────┘ │ job + frame │ │ tables │ └──────────────────┘

6.1 Component Responsibilities

FastAPI (HTTP Layer)

  • Accept video upload or S3 presigned URL
  • Write video bytes to Supabase Storage
  • Publish SQS job message with video_s3_uri
  • Return job_id to client; client polls GET /jobs/{id}
  • Serve completed results from Postgres
  • Code change needed: ~60 lines in new pipeline/api.py

AWS SQS Job Queue

  • Decouples HTTP tier from GPU-bound processing
  • Handles back-pressure: workers consume at their own pace
  • Message visibility timeout = REQUEST_TIMEOUT × max_frames
  • Dead-letter queue for failed jobs (retry cap = 3)
  • Cost: SQS Standard ~$0.40 / million messages

Pipeline Worker (GPU EC2 / ECS)

  • Polls SQS, downloads video from S3, calls run_pipeline()
  • Inserts PipelineOutput dict into Supabase Postgres
  • Deletes SQS message on success; lets it reappear on failure
  • Code change needed: ~80 lines in new pipeline/worker.py
  • Recommended instance: g4dn.xlarge (T4 GPU, 16 GB VRAM) for 7B model; g5.12xlarge (4×A10G) for 72B

vLLM Inference Server

  • OpenAI-compatible API at /v1/chat/completions
  • Pipeline change needed: replace 5 lines inside VLMAnalyzer._call_backend() , no other module changes
  • Supports continuous batching → much higher throughput than Ollama at scale
  • Recommended: separate GPU instance from worker (avoids VRAM contention)

Supabase Storage

  • Raw video upload bucket (private, presigned URLs)
  • HTML report bucket (optionally public per-job)
  • No code changes , storage client wraps standard S3-compatible API

Supabase Postgres

  • jobs table: job_id, status, video_uri, created_at, completed_at, error
  • analyses table: one row per video , stores full PipelineOutput JSON in a jsonb column
  • frames table (optional normalized): one row per analyzed frame for SQL-queryable behavioral timelines
  • Row-level security: jobs scoped to user_id

6.3 Estimated Inference Cost (7B Model Baseline)

At the current Ollama/7B configuration on a g4dn.xlarge (~$0.53/hr on-demand):

  • ~20–35 seconds per VLM call (breed ID + behavioral analysis)
  • ~2–4 VLM calls per 10-second video at 2 s interval + adaptive sampling
  • ~1–2 minutes total wall time per video
  • Rough cost: $0.02–0.04 per video analysis at 7B

With a 72B model on a g5.12xlarge (~$4/hr) and vLLM continuous batching, latency per call drops significantly (vLLM throughput is 5–10× Ollama at scale), but instance cost is higher. The evaluation phase (Weeks 8–10) should include a latency/cost/accuracy tradeoff benchmark to make this decision with data.

8. Open Questions

Q1: Is the 7B model good enough, or do we need 72B before deployment? , The evaluation phase (Weeks 8–10) will answer this directly. If mental state F1 and pain recall meet targets at 7B, we avoid the ~8× GPU cost increase. If not, the switch is a 5-line code change in VLMAnalyzer._call_backend().

Q2: What is the minimum acceptable VLM latency for the product UX? , Current 7B/Ollama performance is ~1–2 minutes per video, which is acceptable for a "submit and return later" async flow. If the product requires near-real-time feedback (<30 s), vLLM with continuous batching on a larger GPU becomes a hard requirement, not a future optimization.

Q3: Should behavioral outputs be smoothed temporally before being shown to users? , Currently each frame is assessed independently. A tag that appears for one frame and disappears may confuse users. EMA smoothing (α=0.3 across consecutive analyzed frames) is already scoped , the question is whether it should be applied at the pipeline level or as a post-processing step in the product layer.

Q4: How should the annotation pipeline be structured at scale? , 200–400 frames is sufficient for v1 evaluation. If fine-tuning requires 2,000+ labeled frames, a formal annotation pipeline (CVAT + inter-rater agreement protocol + veterinary review) needs to be scoped and resourced. This is the primary risk to the Week 15–17 fine-tuning milestone.

Q5: What is the data retention and privacy policy for uploaded dog videos? , The pipeline processes video on the server side. The client needs to define: how long raw videos are stored (Supabase Storage), whether processed frames are retained, and whether videos are used for model training. This affects Supabase storage costs and any applicable privacy regulations.

Q6: Multi-dog scenes , is this in scope before Week 17? , Current policy: take the highest-confidence YOLO detection and ignore others. This is safe for v1 but will produce incorrect results if two dogs are in frame and the wrong one is analyzed. Resolving this requires ByteTrack integration (tracking individual dogs across frames). Recommend keeping out of scope until the single-dog pipeline is validated and deployed.
