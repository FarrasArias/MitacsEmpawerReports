1. Project Recap

The goal of this Mitacs project is to build an end-to-end pipeline that converts user-uploaded dog videos into structured behavioral insights. The system must identify observable behavioral primitives — activity vs. rest, movement intensity, posture classification, and behavioral transitions — and expose them through an inference API that integrates with the company's existing web application.

The scope is deliberately constrained to observable behavioral primitives in this phase. No clinical or emotional/affect claims are made. The long-term vision is for these low-level behavioral signals to serve as building blocks for higher-order affect and wellbeing research in a future phase.

Weeks 1–5 were reserved for an exploration gate: prototype three candidate technical approaches (A, B, C) and make an evidence-based decision on the path forward. This document reports the outcome of that gate and defines the forward architecture.

Primary Deliverables (by Week 16/17):

  • Production-integrated upload → analyze → insights workflow (Supabase Storage + Postgres + job queue + GPU inference service)
  • Behavior detection model v1 with documented metrics, limitations, and model versioning
  • Reproducible training/inference codebase (Docker), evaluation report, and handoff runbooks

2. Exploration Gate Results — Tracks A, B, C

Three prototype approaches were implemented and evaluated during Weeks 4–5. Below is a summary of findings for each track, followed by the decision rationale.

Track A — Pose-First

Keypoints → Behavior

YOLOv8 detection → RTMPose/ViTPose keypoint extraction → LSTM temporal classifier over keypoint trajectories.

  • Interpretable, modular pipeline
  • Leverages StanfordExtra / AP-10K pose datasets
  • Limitation: Remains low-level vision — sensitive to occlusion, breed morphology, and camera angle. Multi-stage error propagation makes the pipeline fragile in a large production pipeline where robustness is critical.

Partial — Low-level only

Track B — Video Classification

End-to-End Clip Model

X3D / SlowFast backbone with multi-head behavior classification heads, operating on 2–4 second clips.

  • Single-stage inference, captures subtle motion patterns
  • Demonstrated on single frames — results were promising
  • Limitation: Data-hungry for fine-tuning; less interpretable; higher compute at inference time

Good results — single frame demo

Track C — Hybrid Baseline

Detection + Rules

YOLOv8 detection → ByteTrack tracking → optical flow / bounding-box velocity features → rule-based classifier.

  • No training required; fully deterministic and debuggable
  • Guaranteed end-to-end delivery baseline
  • Limitation: Accuracy ceiling bounded by hand-crafted rules; coarse behavioral resolution only

Solid baseline — limited ceiling

Criterion Track A (Pose-First) Track B (Video Classif.) Track C (Hybrid)
Robustness Medium — sensitive to occlusion & pose failure Medium-High — end-to-end learned High — rule-based, no data dependency
Behavioral richness Medium — locomotion & posture High — captures appearance + motion Low — coarse motion stats only

Gate Decision: Track A showed promising interpretability but its reliance on multi-stage low-level vision makes it fragile for reliable integration in a large production pipeline. Track B produced good results on single-frame tests and is the strongest candidate for the learned analysis layer. Track C serves as a guaranteed baseline for motion-level primitives and will be retained as the low-level layer of the forward architecture. The proposed path forward integrates all three insights into a new three-layer design (see Section 4).

3. Track D — VLM Video Analyzer (Prototype 4)

In parallel with the exploration gate, a fourth prototype — Track D — was developed as a starting point for the Track C direction taken to its logical conclusion: using a Vision-Language Model (VLM) to perform rich behavioral assessment directly from video frames. This is Prototype 4 of the system and serves as the foundation for the proposed forward architecture.

3.1 What Track D Does

  • Extracts frames from a video at configurable intervals (default: every 2 seconds)
  • Sends each frame to a locally hosted VLM via Ollama (qwen2.5vl:7b in current development environment)
  • The VLM returns structured JSON assessments across 7 behavioral categories, each with a tag and intensity score (0–10)
  • On the first frame, the model also identifies breed, age, coat color, and size
  • Results are aggregated into a self-contained HTML report with timeline charts and a frame-by-frame analysis table

3.2 Current Limitations

  • Model quality: The current development machine uses qwen2.5vl:7b, a compact VLM. Results are indicative but not production-grade. A larger model (e.g., LLaVA-1.6-34B, Qwen2.5-VL-72B, or a fine-tuned specialist) on server-grade GPU hardware is expected to yield substantially better behavioral assessment quality.
  • Frame sampling is static: All frames are sampled at a fixed interval regardless of activity level. An intelligent sampling layer is needed (see Section 4).
  • No dog isolation: The VLM receives the raw frame without first isolating the dog, which can reduce analysis precision in complex scenes.
  • No temporal integration: Each frame is assessed independently; there is no cross-frame context or smoothing.

Despite these limitations, Track D demonstrates the core feasibility of the approach and produces meaningful structured output that can be used for time-series behavioral analysis, as shown in test2_report.html.

4. Proposed Architecture — Three-Layer Video Analyzer

Based on the exploration gate findings, the proposed forward architecture is a three-layer pipeline that combines low-level computer vision for efficiency and robustness, a mid-layer decision engine for adaptive sampling, and a high-level VLM for rich behavioral understanding.

Layer 1
Low-Level Vision
Dog isolation & motion primitives
YOLOv8 dog detection → bounding box crop → optical flow / frame-diff motion score → activity state (moving vs. static), rough posture proxy, motion intensity (low / medium / high)
↓
Layer 2
Decision Engine
Adaptive sampling & routing
If dog is mostly static → reduce sampling rate (fewer VLM calls, lower cost). If dog is moving or a transition is detected → increase sampling rate. Routes the dog crop to Layer 3 with context metadata (motion score, activity state).
↓
Layer 3
VLM Analysis
Rich behavioral assessment via VLM
Cropped dog frame (with context prompt from Layer 2) → VLM → structured JSON output across 7 behavioral categories with intensity scores. Temporal smoothing applied across consecutive assessments.
↓
Output
Structured Data
Structured behavioral timeline + insights
Time-stamped JSON per frame → aggregated summary statistics → HTML/API response with behavioral timelines, dominant states, transition events, and intensity trends.

4.1 Layer 1 — Low-Level Vision Detail

This layer is responsible for isolating the subject and computing motion-level primitives without needing a VLM call. It provides:

  • Dog detection: YOLOv8-nano/small for fast bounding-box detection on every frame
  • Dog crop: Padded crop of the dog region passed to subsequent layers, reducing noise from background
  • Motion score: Per-frame optical flow magnitude or frame-difference score within the dog bounding box
  • Activity state: Thresholded classification — stationary / low motion / high motion
  • Rough posture proxy: Bounding box aspect ratio as a proxy for standing vs. lying (fast heuristic, not a trained model)

4.2 Layer 2 — Decision Engine Detail

The decision engine controls when to invoke the expensive VLM layer and what context to supply. Key logic:

  • If dog is stationary for ≥ N consecutive seconds → sample at reduced rate (e.g., 1 frame / 5s)
  • If motion score exceeds threshold → sample at full rate (e.g., 1 frame / 1s)
  • On activity-state transition detection → force an immediate sample
  • Prepends motion context to the VLM prompt: "The dog is currently moving at high intensity. Focus on gait, weight distribution, and any signs of discomfort."

4.3 Layer 3 — VLM Analysis Detail

The VLM receives the dog crop (not the full frame) plus a context-enriched prompt and returns a structured JSON response conforming to the schema already established in Track D (schema.py). The seven behavioral categories assessed are:

5. Structured Outputs & Time Analysis

The pipeline produces a structured JSON output per video that can be consumed by any downstream system (web app, reporting dashboard, research tools). A reference HTML report (test2_report.html) demonstrates the kind of insights derivable from this structured data.

5.1 Per-Frame Output Schema

// Per-frame structured output (one entry per sampled frame) { "timestamp_seconds": 4.0, "frame_index": 2, "layer1": { "motion_score": 0.12, // optical flow magnitude "activity_state": "stationary", "posture_proxy": "lying", // bbox aspect ratio heuristic "dog_bbox": [120, 80, 540, 410] }, "eyes_and_gaze": { "tag": "soft gaze", "intensity": 4 }, "ear_position": { "tag": "ears neutral", "intensity": 3 }, "mouth_and_muzzle": { "tag": "mouth relaxed open", "intensity": 6 }, "body_posture": { "tag": "posture relaxed", "intensity": 7 }, "tail_carriage": { "tag": "tail neutral", "intensity": 5 }, "mental_state": { "tag": "relaxed", "intensity": 7 }, "pain_indicators": { "tag": "no obvious pain signs", "intensity": 0 }, "notes": "Dog appears calm and comfortable in a relaxed lying position." }

5.2 Video-Level Aggregated Output

Per-frame outputs are aggregated to produce a video-level summary:

  • Dominant state per category: Most frequent tag + percentage of frames
  • Average intensity per category over the video duration
  • Activity timeline: Time-series of activity state (stationary / moving) with timestamps
  • Transition events: Detected posture/activity transitions with timestamps and durations
  • Behavioral intensity trends: Per-category intensity over time
  • Breed identification: Breed, confidence, approximate age, size category

7. Open Questions

Q1: Fine-tuning the VLM on domain-specific data
The biggest quality boost would come from fine-tuning the VLM on a labeled dataset of dog behavior videos with verified frame-level behavior annotations. This only becomes viable once a high-quality data collection + annotation pipeline exists and the dataset is large enough (early estimate: thousands of labeled clips). It’s a real option, but should be revisited in Weeks 9–11.

Q2: Optimal output schema for the product
The current seven-category schema was built for veterinary behavioral assessment, but it’s still unclear what signals the product UI should show and how (raw tags, plain-language summaries, wellness scores, etc.). This needs product-team input and iteration based on early user feedback.

Q3: Model selection and hardware budget
The best production VLM depends on the GPU inference server hardware budget. A larger model (72B) should give better behavioral assessments but will cost more to run, while a smaller fine-tuned model could match or beat it in this domain for less. Once server hardware is confirmed, a head-to-head evaluation on a real video test set is needed.

Q4: Adaptive sampling threshold calibration
The Layer 2 motion thresholds that control when sampling rate increases or decreases must be calibrated empirically on representative videos. If thresholds are too aggressive, VLM calls are wasted; if too conservative, important behavior moments get missed.

Q5: Multi-dog scenes
The current architecture assumes single-dog videos. For multi-dog scenes, the pipeline would need a tracking layer to maintain consistent dog IDs and crop each dog separately. This is out of scope for v1, but should be documented as a known limitation.

Q6: Annotation and ground truth
Quantitative evaluation (F1, precision, recall) requires a ground-truth labeled validation set. The Week 3 label taxonomy exists, but the annotation tooling and pipeline still need to be built, making this the primary risk to the Week 11 milestone.
